## 二、有待改进的工作流程

### 2.1 当前流程：差异表达分析报告撰写

在日常转录组学分析工作中，完成 RNA-seq 数据的差异表达分析后，需要撰写结构化的分析报告。当前工作流程如下：

```
┌─────────────┐    ┌─────────────┐    ┌──────────────┐    ┌──────────────┐    ┌─────────────┐
│ DESeq2/edgeR │───>│  结果文件    │───>│  人工浏览    │───>│  手动绘图    │───>│ 手动撰写    │
│ 差异分析     │    │  (CSV/TSV)  │    │  结果数据    │    │  火山图/MA图 │    │ 分析报告    │
└─────────────┘    └─────────────┘    └──────────────┘    └──────────────┘    └─────────────┘
       ↓                   ↓                  ↓                   ↓                   ↓
   已自动化            已有数据          ⚠️ 耗时费力          ⚠️ 重复劳动         ⚠️ 主观性强
   (Snakemake)                         (打开多个文件,       (每项目重复写        (格式不统一,
                                        人工统计汇总)        相似的绑图代码)      质量参差不齐)
```

**当前流程痛点**：

1. **人工统计汇总耗时**：分析人员需要手动打开 CSV 文件，统计显著差异基因数量（上调/下调）、log2FC 范围、Top 基因列表等。每个项目约需 20-30 分钟。
2. **可视化图表重复绘制**：火山图、MA 图、分布图等是每份报告的标配，但每次都需要重新编写或调整绑图代码。每个项目约需 15-20 分钟。
3. **报告撰写主观性强**：分析人员根据个人经验撰写结果解读和讨论部分，不同人员的报告质量和风格差异大。每个项目约需 40-60 分钟。
4. **格式不统一**：不同项目、不同分析人员产出的报告格式和详细程度不一，不利于团队协作和结果复用。

**时间成本估算**：每份报告约需 **1.5-2 小时**，每月约产生 10-15 份报告，月均占用约 **20-30 小时**。

### 2.2 流程中的 AI 可替代部分

| 步骤 | 当前方式 | AI 替代方案 | 替代可行性 |
|------|---------|------------|-----------|
| 统计汇总 | 人工打开文件统计 | Python 自动化脚本 | ★★★★★ 完全替代 |
| 可视化绘图 | 手动编写 matplotlib 代码 | 封装好的可视化函数 | ★★★★★ 完全替代 |
| 结果解读 | 人工阅读文献 + 经验判断 | LLM 基于统计结果自动生成 | ★★★★☆ 大部分替代 |
| 报告排版 | 手动编辑 Word/Markdown | 模板化自动生成 | ★★★★★ 完全替代 |

### 3.2 模块设计

本项目采用模块化设计，各模块职责明确：

#### 模块 1：数据加载模块 (`load_deg_results`)
- 功能：读取 DESeq2/edgeR 输出的 CSV 差异表达结果
- 输入：CSV 文件路径
- 输出：Pandas DataFrame
- 容错：自动检测必要列（log2FoldChange, pvalue, padj）

#### 模块 2：统计分析模块 (`run_statistical_analysis`)
- 功能：根据阈值筛选差异基因，计算统计指标
- 输入：DataFrame + 阈值参数
- 输出：结构化统计摘要（基因总数、显著差异基因数、上调/下调分类、Top 基因列表）
- 参数：fc_threshold（默认 1.0）、pval_threshold（默认 0.05）

#### 模块 3：可视化模块 (`plot_volcano`, `plot_ma`, `plot_distribution`)
- 功能：自动生成火山图、MA 图、分布图
- 特点：自动标注 Top 差异基因，支持中文标签，统一配色方案

#### 模块 4：LLM 报告生成模块 (`call_llm`, `generate_report`)
- 功能：调用大语言模型（LLM）自动撰写分析报告
- 支持的 LLM 后端：OpenAI API 及所有兼容接口（DeepSeek、智谱 GLM、通义千问等）
- 降级策略：当 API 不可用时，自动切换为离线模板模式

#### 模块 5：主流程 (`run_agent`)
- 功能：串联上述模块，形成完整的 Agent Pipeline
- 输出：Markdown 报告 + PNG 图表 + JSON 统计数据

### 4.2 LLM 集成方式

采用 OpenAI 兼容接口标准，通过环境变量配置：

```python
# 支持 OpenAI / DeepSeek / 智谱 / 通义千问 等
api_key = os.environ.get("LLM_API_KEY")
base_url = os.environ.get("LLM_BASE_URL", "https://api.openai.com/v1")
model = os.environ.get("LLM_MODEL", "gpt-4o-mini")
```

**Prompt 工程**：
- System Prompt：设定角色为"资深生物信息学分析师"，指定学术正式风格
- User Prompt：将统计分析结果（基因数、阈值、Top 基因列表）结构化注入
- Temperature：0.3（偏低，确保输出稳定性和专业性）

### 4.3 降级策略

当 LLM API 不可用时（未配置 API Key、网络问题等），自动降级为离线模板模式，确保报告的基本结构完整，用户仍可获得统计数据和可视化图表。

## 六、讨论与展望

### 6.1 优势

1. **效率提升显著**：将报告生成时间从 1-2 小时缩短至分钟级，释放分析人员的精力用于更有价值的科研工作。
2. **质量一致性好**：统一的统计口径和报告格式，避免了人为差异。
3. **易于集成**：命令行接口设计简洁，可方便地集成到现有的 Snakemake 流程中。
4. **LLM 增强**：利用大语言模型的语言理解能力，生成专业、有逻辑的分析报告。

### 6.2 局限性

1. **依赖标准格式**：要求输入为 DESeq2/edgeR 标准输出格式，兼容性有待扩展。
2. **LLM 幻觉风险**：大语言模型可能生成不准确的功能描述，需要人工审核。
3. **功能分析缺失**：当前版本尚未集成 GO/KEGG 富集分析，计划在后续版本中加入。

### 6.3 后续改进方向

1. **集成功能富集分析**：调用 clusterProfiler 或 g:Profiler API 自动完成 GO/KEGG 富集。
2. **多格式输出**：支持 Word（.docx）和 PDF 输出，满足不同场景需求。
3. **交互式报告**：基于 Plotly/Dash 生成交互式 HTML 报告。
4. **多组学扩展**：支持单细胞、ATAC-seq、蛋白质组等其他组学数据的报告生成。
5. **RAG 增强**：构建本地文献知识库，结合 RAG 技术提升 LLM 解读的准确性。
